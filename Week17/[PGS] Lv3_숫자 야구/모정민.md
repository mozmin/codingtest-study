# [Level 2] 숫자 야구

[문제 링크](https://school.programmers.co.kr/learn/courses/30/lessons/451808)

`#구현` `#브루트포스(완전탐색)` `#소거법`

### 제출 일자
2026년 6월 11일

### 문제 설명 (요약)
1부터 9까지의 서로 다른 숫자 4개로 구성된 비밀번호를 맞추는 숫자 야구 게임입니다. 우리에게는 정답 여부를 판단하고 힌트("xS yB")를 제공하는 `submit` 함수(리모컨)가 주어집니다. 단, 제출 횟수 제한 `n`이 최대 3024회에서 **최소 6회**까지 줄어들기 때문에, 최소한의 제출만으로 정답을 찾아내야 합니다.

---

### 핵심 풀이 전략
1. **전체 후보군 생성**: 1000~9999 사이의 숫자 중 0이 포함되지 않고 4자리가 모두 서로 다른 유효한 숫자 야구 정답 후보 리스트를 전부 뽑아둡니다. (총 $9 \times 8 \times 7 \times 6 = 3,024$개)
2. **실제 힌트 획득**: 남아있는 후보 중 하나를 골라 `submit.apply(guess)`를 호출해 실제 게임기의 힌트(`"xS yB"`)를 얻습니다.
3. **가상 시뮬레이션을 통한 소거법 (핵심)**: 실제 리모컨은 6번밖에 못 누르므로, 머릿속(코드)으로 가상 숫자 야구를 돌립니다. 남아있는 3,024개의 후보를 돌며 **'만약 이 후보가 진짜 정답이라면, 방금 내가 던진 추측값과 비교했을 때 똑같은 힌트가 나올까?'**를 계산하여, 결과가 일치하지 않는 가짜 후보들을 리스트에서 전부 지워나가며 정답을 압축합니다.

---

### 정답 코드 (리팩토링 - 가독성 및 가상 판정 로직 최적화)
배열을 매번 새로 생성하는 비용을 줄이기 위해 숫자를 다루는 파싱 부분을 가볍게 유지하고, 전체 흐름을 명확하게 분리한 최적화 구조입니다.

```java
import java.util.*;
import java.util.function.Function;

class Solution {
    public int solution(int n, Function<Integer, String> submit) {
        List<Integer> candidates = generateCandidates();

        while (candidates.size() > 1) {
            int guess = candidates.get(0);
            String actualHint = submit.apply(guess);
            
            if (actualHint.equals("4S 0B")) return guess;

            List<Integer> filtered = new ArrayList<>();
            for (int candidate : candidates) {
                // 내 머릿속 가상 대결(candidate vs guess) 결과가 실제 힌트와 같다면 후보 유지
                if (checkVirtualHint(candidate, guess).equals(actualHint)) {
                    filtered.add(candidate);
                }
            }
            candidates = filtered;
        }

        return candidates.isEmpty() ? 0 : candidates.get(0);
    }

    private List<Integer> generateCandidates() {
        List<Integer> list = new ArrayList<>();
        for (int i = 1234; i <= 9876; i++) {
            int[] d = { i / 1000, (i / 100) % 10, (i / 10) % 10, i % 10 };
            if (d[0] == 0 || d[1] == 0 || d[2] == 0 || d[3] == 0) continue;
            if (d[0] != d[1] && d[0] != d[2] && d[0] != d[3] && d[1] != d[2] && d[1] != d[3] && d[2] != d[3]) {
                list.add(i);
            }
        }
        return list;
    }

    private String checkVirtualHint(int candidate, int guess) {
        int[] cArr = { candidate / 1000, (candidate / 100) % 10, (candidate / 10) % 10, candidate % 10 };
        int[] gArr = { guess / 1000, (guess / 100) % 10, (guess / 10) % 10, guess % 10 };

        int s = 0, b = 0;
        for (int i = 0; i < 4; i++) {
            for (int j = 0; j < 4; j++) {
                if (cArr[i] == gArr[j]) {
                    if (i == j) s++;
                    else b++;
                }
            }
        }
        return s + "S " + b + "B";
    }
}
```

### 회고 및 삽질기록 🧠
- 이 문제는 풀지 못해서 AI의 힘을 빌렸다. 그래도 여전히 이해가 안간다...
- 일반적인 코딩 테스트 문제와 달리 파라미터로 함수(Function<Integer, String> submit)가 직접 넘어와서 처음 문제를 마주했을 때 매개변수의 역할과 문제 자체를 이해하는 데 크게 애를 먹었다.

- 단순히 1000부터 9999까지 다 찔러보는 완전 탐색을 돌리면 효율성 제한(n = 6) 조건에서 무조건 터진다는 것을 알았다. 인터랙티브(Interactive)하게 질문 제한 수가 정해져 있는 문제는 질문 1번으로 얻은 힌트를 바탕으로 전체 후보를 필터링하는 '소거법 알고리즘'을 구축해야 함을 깨달았다.

- 가상 시뮬레이션용 판정 메서드(checkVirtualHint)를 따로 두고 원본 데이터와 비교하는 발상 전환이 핵심 열쇠였다. 구현이 복잡해 보일수록 문제를 작은 기능(후보 생성, 실제 힌트 획득, 가상 힌트 비교 및 소거)으로 나누어 설계하는 습관을 지녀야겠다.
