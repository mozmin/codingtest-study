# [Level 1] 노란불 신호등

### 문제 설명 (요약)
모든 신호등이 초록불 → 노란불 → 빨간불 순서로 무한히 반복되며, 1초부터 시작해 처음에는 모두 초록불 상태입니다. 모든 신호등이 동시에 노란불이 되는 가장 빠른 시각(초)을 구하는 문제입니다. 만약 모든 신호등이 동시에 노란불이 되는 경우가 존재하지 않는다면 `-1`을 반환합니다.

---

### 핵심 풀이 전략
1. **시뮬레이션 및 시간 탐색**: 1초부터 시작하여 시각을 1씩 증가시키며 모든 신호등의 상태를 확인합니다.
2. **나머지 연산을 이용한 주기성 판단**: 각 신호등의 총 주기(`G + Y + R`)를 구한 뒤, 현재 시각을 주기로 나눈 나머지(`time % cycle`)를 이용해 현재 신호등이 노란불 영역에 속하는지 판별합니다.
3. **탐색 범위(상한선) 설정**: 모든 신호등이 노란불이 될 수 없는 경우(`-1`을 반환해야 하는 경우) 무한 루프에 빠지지 않도록 적절한 최대 탐색 시간을 지정해야 합니다. 신호등의 개수와 주기의 최댓값을 고려하여 충분히 큰 범위까지 탐색합니다.

---

### 내가 구현한 구현 코드
```java
import java.util.*;

class Solution {
    public int solution(int[][] signals) {
        int answer = 0;
        
        // 1초부터 시작
        for (int i = 1; i <= Math.pow(20, signals.length); i++){
            
            int count = 0;
            
            for (int j = 0; j < signals.length; j++){
                if (isYellow(signals, j, i)) count++;
            }
            
            if (count >= signals.length){
                answer = i;
                break;
            }
        }
        
        return answer == 0 ? -1 : answer;
        
    }
    
    public boolean isYellow(int[][] signals, int index, int time){
        
        int cycle = signals[index][0] + signals[index][1] + signals[index][2];
        
        int rest = time % cycle;
        
        if (signals[index][0] < rest && rest <= (signals[index][0] + signals[index][1])){
            return true;
        }
        
        return false;
    }
    
}
```

## 정답 코드 (리팩토링 - 탐색 범위 최적화 및머지 조건 예외 처리)
나머지 연산(`time % cycle`)을 수행할 때 시각이 주기의 배수인 경우(`rest == 0`)에 대한 예외 처리를 명확히 하거나, `rest` 범위를 직관적으로 개선해야 합니다. 또한 정답이 없는 케이스를 빠르게 거르기 위해 탐색 상한선을 안전하고 명확하게 수치화하여 설정합니다.

```java
import java.util.*;

class Solution {
    public int solution(int[][] signals) {
        // 주기의 최댓값 조건과 신호등 개수를 고려하여 충분히 큰 안전 상한선 설정
        // 혹은 모든 신호등 주기의 최소공배수(LCM) 시점까지 탐색하도록 설정 가능
        int maxLimit = 1000000; 

        for (int time = 1; time <= maxLimit; time++) {
            boolean allYellow = true;
            
            for (int[] signal : signals) {
                if (!isYellowSignal(signal, time)) {
                    allYellow = false;
                    break; // 한 번이라도 노란불이 아니면 정체할 필요 없이 조기 종료
                }
            }
            
            if (allYellow) {
                return time; // 가장 빠른 시각이므로 즉시 반환
            }
        }
        
        return -1; // 상한선까지 모든 신호등이 노란불이 되는 타이밍이 없으면 -1
    }
    
    private boolean isYellowSignal(int[] signal, int time) {
        int g = signal[0];
        int y = signal[1];
        int r = signal[2];
        int cycle = g + y + r;
        
        // 1부터 시작하는 시각을 0-indexed로 맞추기 위해 (time - 1) 사용
        int currentPos = (time - 1) % cycle;
        
        // 초록불 구간: 0 ~ g-1
        // 노란불 구간: g ~ g+y-1
        return currentPos >= g && currentPos < g + y;
    }
}
```

## 회고 및 삽질기록 🧠
- **막연한 탐색 범위 설정의 위험성**: `Math.pow(20, signals.length)`와 같이 임의로 대략적인 범위를 지정했다가, 최소공배수 관계에 의해 정답 시각이 주기를 훌쩍 뛰어넘는 반례 케이스를 놓칠 뻔했다. 문제의 제한 사항을 보고 주기의 결합으로 표현될 수 있는 수치의 최대 상한선을 논리적으로 계산하거나, 연산 횟수가 1억 번을 넘지 않는 선에서 넉넉한 상한선(예: 1,000,000)을 명시적으로 잡는 버릇을 들여야겠다.
- **주기성과 나머지 연산의 삐끗함**: 시각이 1초부터 시작하기 때문에 `time % cycle`을 그대로 사용하면 딱 나누어 떨어지는 순간(`rest == 0`)이 주기의 마지막(빨간불의 마지막 초)임에도 불구하고 조건문에서 오작동을 일으키기 쉽다. 이를 방지하기 위해 전체 시각 체계를 `time - 1`로 두어 `0-indexed`로 통일한 뒤 각 구역 범위를 `[g, g + y)` 형태로 깔끔하게 비교하는 것이 조건식 실수를 줄이는 지름길임을 깨달았다.
- **조기 종료(Pruning) 누락**: 모든 신호등이 만족하는지 확인할 때 `count` 변수를 두고 끝까지 조사하기보다, 단 하나의 신호등이라도 노란불이 아니라면 즉시 다음 `time`으로 넘어가는 `break` 문을 추가하여 불필요한 연산을 아낄 수 있도록 가독성과 성능을 동시에 챙겨야겠다.
