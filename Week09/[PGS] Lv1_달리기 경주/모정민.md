# [Lv 1] 달리기 경주

[문제 링크](https://school.programmers.co.kr/learn/courses/30/lessons/178871)

`#해시` `#배열` `#구현`

### 제출 일자

### 문제 설명 (요약)
선수들의 이름이 1등부터 순서대로 담긴 배열 `players`와 해설진이 부른 이름이 담긴 배열 `callings`가 주어짐.
이름이 불린 선수는 자신의 바로 앞 선수를 추월한 것이며, 경주가 끝났을 때 선수들의 이름을 1등부터 순서대로 반환하는 문제.

- **players**: 선수 명단 (최대 50,000명)
- **callings**: 호출된 이름 (최대 1,000,000번)

### 입력
- `players`: 현재 등수순 선수 명단
- `callings`: 추월한 선수들의 이름

### 출력
- 경주 종료 후의 최종 선수 명단

---

### 핵심 풀이 전략
1. **시간 복잡도 최적화 ($O(N \times M)$ 방지)**
   - `ArrayList.indexOf()`나 `remove()`를 사용하면 매 호출마다 리스트를 순회하므로 전체 연산량이 약 500억 번에 달해 시간 초과가 발생.
   - 따라서 **조회 시간 $O(1)$**을 보장하는 `HashMap`을 도입.

2. **데이터 이중 관리 (Map & Array)**
   - **`Map<String, Integer> rankMap`**: "선수 이름"을 Key로, "현재 등수"를 Value로 저장.
   - **`String[] players`**: 실제 등수 순서를 유지하며, 인덱스를 통해 선수 이름을 즉시 교체.

3. **위치 교환(Swap) 프로세스**
   - 호출된 선수(`calling`)의 현재 등수를 Map으로 가져옴.
   - `players` 배열에서 해당 선수와 그 바로 앞 선수(`prevPlayer`)의 위치를 서로 바꿈.
   - 위치가 바뀐 **두 선수 모두**의 등수 정보를 Map에서도 업데이트.

---

### Java 구현 코드

```java
import java.util.*;

class Solution {
    public String[] solution(String[] players, String[] callings) {
        
        ArrayList<String> list = new ArrayList<>();
        Map<String, Integer> map = new HashMap<>();
        
        // 선수 목록 저장
        for(int i = 0; i < players.length; i++){
            map.put(players[i], i);
        }
        
        for(String c : callings){
            
            int index = map.get(c);
            map.put(c, map.getOrDefault(c, 0) -1);
            
            String front = players[index-1];
            
            map.put(front, map.getOrDefault(front, 0) +1);
            
            players[index] = front;
            players[index-1] = c;
        }
        
        return players;
    }
}
```

### 삽질 기록 🧠
1. 처음에는 ArrayList가 다루기 편해서 선택했지만, 내부적으로 indexOf()가 $O(N)$임을 간과했다. 데이터가 5만 개가 넘어가니 선형 탐색의 한계가 명확히 드러났다.

### 배운 점 && 보완할 점
1. 자료구조의 조합: 순서 유지(Array)와 빠른 탐색(Map)을 결합하는 방식은 실무에서도 자주 쓰이는 최적화 기법임을 배웠다.
2. 제약 조건 읽기: 입력값의 범위를 먼저 보고 $O(1)$ 접근이 필요한지 판단하는 능력이 얼마나 중요한지 체감했다.
3. 원본 배열 players[]의 순서만 바꾸고 Map의 데이터를 갱신하지 않으면, 다음 추월 시 엉뚱한 위치를 참조하게 된다. 데이터가 두 곳에서 관리될 때는 반드시 양쪽 모두를 업데이트해야 함을 다시 깨달았다.
