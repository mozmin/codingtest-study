# [Medium] Coin Change

[문제 링크](https://leetcode.com/problems/coin-change/)

`#동적계획법(DP)` `#배열` `#너비우선탐색(BFS)`

### 제출 일자 
2026년 6월 25일 12ms

### 문제 설명 (요약)
다양한 금액의 동전 종류들을 나타내는 정수 배열 `coins`와 총액 `amount`가 주어질 때, 그 금액을 만들 수 있는 **최소한의 동전 개수**를 구하는 문제입니다. 만약 어떤 동전 조합으로도 해당 금액을 만들 수 없다면 `-1`을 반환합니다. 각 종류의 동전은 무한정 사용할 수 있다고 가정합니다.

---

### 핵심 풀이 전략
1. **그리디(Greedy)의 한계와 DP의 필요성**: 가장 큰 단위의 동전부터 무조건 많이 고르는 그리디 방식은 반례가 존재합니다. 예를 들어 `coins = [1, 6, 7, 9, 11]`, `amount = 13`일 때, 그리디는 `11 + 1 + 1`로 3개를 고르지만, 최적해는 `6 + 7`로 2개입니다. 따라서 모든 경우를 고려하되 중복 계산을 방지하는 **동적계획법(DP)**을 사용해야 합니다.
2. **DP 테이블 정의 및 초기화**: `dist[i]`를 금액 `i`를 만드는 데 필요한 최소 동전 개수로 정의합니다. 배열 크기는 `amount + 1`로 선언하고, 최솟값 비교를 위해 만들 수 없는 가상의 큰 값인 `amount + 1`로 채워둡니다. 금액 0원을 만드는 데 필요한 동전은 0개이므로 `dist[0] = 0`으로 초기화합니다.
3. **점화식 도출 (Bottom-up)**: 1원부터 `amount`원까지 순차적으로 채워나갑니다. 금액 `i`를 만들 때, 가지고 있는 동전 `c`를 하나씩 대입해 보며 `i - c`원이 만들어질 수 있는 상태라면 아래 점화식을 적용합니다.
   $$dist[i] = \min(dist[i], dist[i - c] + 1)$$

---

### 내가 구현한 구현 코드
```java
import java.util.*;

class Solution {
    public int coinChange(int[] coins, int amount) {
        if (amount == 0) return 0;
        
        int[] dist = new int[amount + 1];
        Arrays.fill(dist, amount + 1);
        dist[0] = 0; // 0원을 만드는 데 필요한 동전은 0개
        
        for (int i = 1; i <= amount; i++) {
            for (int c : coins) {
                if (i - c >= 0) {
                    dist[i] = Math.min(dist[i], dist[i - c] + 1);
                }
            }
        }
        
        if (dist[amount] == amount + 1) return -1;
        else return dist[amount];
    }
}
```

## 회고 및 삽질기록 🧠
- 동전 거스름돈 문제는 얼핏 직관적으로 그리디 알고리즘이 먼저 떠오르기 쉬우나, 동전들 간의 관계가 배수 관계(ex: 10원, 50원, 100원)가 아닌 무작위(ex: 1원, 6원, 7원)일 때는 무조건 DP나 완전탐색 계열로 접근해야 안전하다는 점을 다시금 깨달았다.
- 초기화 값으로 `Integer.MAX_VALUE`를 무심코 사용했다가, 점화식의 `dp[i - coin] + 1` 연산 과정에서 정수 오버플로우(Overflow)가 발생해 음수 최솟값으로 꼬이는 삽질을 할 뻔했다. 가상의 무한대를 설정할 때는 문제의 최대 정답 범위를 고려해 `amount + 1` 정도로 안전하게 잡아주는 테크닉이 유용함을 배웠다.
- 루프의 순서(금액 중심 vs 동전 중심)에 따라 조건문 하나를 통째로 줄일 수 있다는 점을 리팩토링 과정을 통해 체감했다. 탑다운(Top-down) 재귀보다는 바텀업(Bottom-up) 반복문이 직관적이고 효율적인 문제였다.
