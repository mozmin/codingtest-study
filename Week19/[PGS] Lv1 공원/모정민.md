### 문제 설명 (요약)
지민이가 가진 다양한 크기의 정사각형 돗자리 중 공원의 빈 공간에 깔 수 있는 **가장 큰 돗자리의 한 변 길이**를 구하는 문제입니다. 공원의 자리 배치도가 2차원 문자열 배열 `park`로 주어지며, 빈 공간은 `"-1"`, 사람이 있는 공간은 알파벳으로 표시됩니다. 지민이가 가진 돗자리 크기 배열 `mats`가 주어질 때, 다른 사람과 겹치지 않고 공원 구역을 벗어나지 않으면서 깔 수 있는 가장 큰 돗자리의 변의 길이를 반환합니다. 어떤 돗자리도 깔 수 없다면 `-1`을 반환합니다.

---

### 핵심 풀이 전략
1. **역순 정렬을 통한 최적화 (Greedy 접근)**: 가장 큰 돗자리부터 깔 수 있는지 확인하고, 성공하면 즉시 그 크기를 반환하는 것이 효율적입니다. 이를 위해 `mats` 배열을 오름차순 정렬한 뒤 역순(내림차순)으로 배치한 새로운 배열을 활용해 큰 크기부터 순회합니다.
2. **2차원 배열 전수 조사**: 공원의 모든 좌표 `(i, j)`를 시작점으로 삼아 돗자리를 놓을 수 있는지 탐색합니다. 단, 시작점의 원소가 `"-1"`인 경우에만 돗자리 배치를 시도(`check`)합니다.
3. **정사각형 영역 검증 및 경계 조건 처리**: 선택한 크기 `m`만큼의 정사각형 영역 `(i ~ i+m-1, j ~ j+m-1)`을 돌며 공원 범위를 벗어나지 않는지(`a >= park.length` 또는 `b >= park[0].length`), 그리고 도중에 사람이 없는지(`!park[a][b].equals("-1")`)를 철저히 검증합니다.

---

### 내가 구현한 구현 코드
```java
import java.util.*;

class Solution {
    
    int answer;
    
    public int solution(int[] mats, String[][] park) {
        
        answer = -1;
        Arrays.sort(mats);
        int[] newMat = new int[mats.length];
        
        for(int i = 0; i < mats.length; i++){
            newMat[i] = mats[mats.length-i-1];
        }
        
        for(int m : newMat){
            for(int i = 0; i < park.length; i++){
            
                for(int j = 0; j < park[0].length; j++){
                
                    // 만약 -1이라면
                    if(park[i][j].equals("-1")){
                        check(i, j, park, m);
                    }
                    
                    if(answer != -1) return answer;
                }
            }
        }
        return answer;
        
    }
    
    public void check(int i, int j, String[][] park, int m){
        
        for(int a = i; a < i+m; a++){
            for(int b = j; b < j+m; b++){
                
                if(a >= park.length || b >= park[0].length || !park[a][b].equals("-1")) return;
                
            }
        }
        
        answer = m;
        
    }
    
}
```

## 정답 코드 (리팩토링 - 전역 변수 제거 및 조기 종료 로직 가독성 개선)
전역 변수(`static` 또는 클래스 멤버 변수)를 사용하면 상태 관리가 까다로워지고 테스트 케이스 간에 값이 꼬일 위험이 있습니다. `check` 메서드가 돗자리를 깔 수 있는지 여부를 `boolean`으로 반환하도록 변경하고, 깔 수 있다면 즉시 해당 크기 `m`을 반환(Early Return)하게 만들면 구조가 훨씬 깔끔해집니다.

```java
import java.util.*;

class Solution {
    public int solution(int[] mats, String[][] park) {
        // 돗자리 크기를 오름차순 정렬
        Arrays.sort(mats);
        
        // 가장 큰 돗자리 크기부터 역순으로 탐색
        for (int i = mats.length - 1; i >= 0; i--) {
            int m = mats[i];
            
            // 공원 전체를 순회하며 돗자리를 놓을 시작점 탐색
            for (int r = 0; r < park.length; r++) {
                for (int c = 0; c < park[0].length; c++) {
                    // 시작 구역이 빈 공간이고, m x m 크기로 돗자리를 깔 수 있다면
                    if (park[r][c].equals("-1") && canPlace(r, c, park, m)) {
                        return m; // 가장 큰 크기부터 찾았으므로 발견 즉시 반환
                    }
                }
            }
        }
        
        return -1; // 깔 수 있는 돗자리가 없는 경우
    }
    
    // (r, c) 좌표를 좌측 상단 시작점으로 하여 m x m 크기의 돗자리를 깔 수 있는지 검증
    private boolean canPlace(int r, int c, String[][] park, int m) {
        // 돗자리가 공원 범위를 벗어나는지 미리 체크
        if (r + m > park.length || c + m > park[0].length) {
            return false;
        }
        
        // m x m 영역 내부 검증
        for (int i = r; i < r + m; i++) {
            for (int j = c; j < c + m; j++) {
                // 한 칸이라도 빈 공간("-1")이 아니라면 설치 불가
                if (!park[i][j].equals("-1")) {
                    return false;
                }
            }
        }
        
        return true;
    }
}
```

## 회고 및 삽질기록 🧠
- 클래스 멤버 변수(`answer`)에 의존하여 헬퍼 메서드 안에서 전역 상태를 바꾸는 방식을 사용했다. 정답을 맞히는 데는 문제가 없었지만, 이 방식은 가독성을 떨어뜨리고 부수 효과(Side Effect)를 낳기 쉽다. 검증 메서드는 참/거짓(`boolean`)만 명확히 반환하고, 실제 정답 처리는 메인 제어 흐름에서 담당하도록 분리하는 습관을 지녀야겠다.
- 배열을 역순으로 다루기 위해 새로운 `newMat` 배열을 파고 반복문을 한 번 더 돌렸는데, 기존 `mats`를 `Arrays.sort()`한 뒤에 인덱스를 역순(`mats.length - 1`부터 `0`까지)으로 접근하면 추가 배열 생성 없이 깔끔하게 메모리와 연산을 아낄 수 있다는 점을 간과했다.
- `check` 메서드 내부에서 범위를 벗어나는 조건문(`a >= park.length` 등)을 루프 내부에서 매번 검사하는 것보다, 2중 루프에 진입하기 전에 `r + m > park.length` 형태로 미리 필터링(Pruning)해 주면 불필요한 반복 횟수를 훨씬 줄일 수 있음을 배웠다.
