### 문제 설명 (요약)
어떤 과학자가 발표한 논문 `n`편 중, `h`번 이상 인용된 논문이 `h`편 이상이고 나머지 논문이 `h`번 이하 인용되었다면 `h`의 최댓값이 H-Index가 됩니다. 발표한 논문의 인용 횟수가 담긴 배열 `citations`가 주어질 때 H-Index를 구하는 문제입니다.

---

### 핵심 풀이 전략
1. **정렬을 통한 비교 최적화**: 배열을 오름차순으로 정렬하면, 특정 위치의 인용 횟수(`citations[i]`)를 확인할 때 그 뒤에 있는 논문들은 모두 그보다 크거나 같음을 보장할 수 있습니다.
2. **H-Index의 조건식 변환**: `n - i`는 현재 논문을 포함하여 인용 횟수가 `citations[i]` 이상인 논문의 총 개수입니다. 따라서 `citations[i] >= n - i`가 성립하는 순간의 `n - i`가 조건을 만족하는 H-Index의 최댓값이 됩니다.

---

### 내가 구현한 구현 코드 (오답 - 테스트 11 실패)
```java
import java.util.*;

class Solution {
    public int solution(int[] citations) {
        int answer = 0;
        
        Arrays.sort(citations);
        int arrLength = citations.length;
        
        // h 0 ~ 1000
        outer: for(int i = 0; i < arrLength; i++){
            
            int count = arrLength - i;
            
            if(citations[i] >= count){
                return count;
            }
        }
        return 0;
    }
}
```


## 회고 및 삽질기록 🧠
- **엣지 케이스 (테스트 11 실패 원인)**: `[3, 3, 3, 3, 3]`과 같은 반례가 들어왔을 때 오작동했다. H-Index의 정의에 따라 3번 이상 인용된 논문이 3편 이상(5편)이므로 H-Index는 3이 되어야 한다. 하지만 기존 코드는 `arrLength - count`(5)가 `i`(3)와 **정확히 일치하지 않아** 조건을 무시하고 루프를 빠져나가 0을 반환해버렸다. 인용 횟수와 논문 편수가 반드시 딱 맞아떨어지지 않아도 `h` 조건을 충족할 수 있음을 간과했다.
- **문제 조건 꼼꼼히 읽기**: "h편 이상"이라는 텍스트를 코드화할 때 "h편과 동일하다(`==`)"로 무의식적으로 치환해서 풀이하는 논리적 오류를 범했다. 문제의 의도를 벗어나 자의적으로 좁혀서 해석하지 않도록 주의가 필요하다.
- **시간 복잡도 최적화**: 0부터 1000까지 임의의 상수 범위로 이중 반복문(`O(N^2)` 형태)을 돌기보다는, 배열을 정렬한 특성을 이용해 순회 한 번(`O(N)`)으로 깔끔하게 처리하는 접근법이 훨씬 안전하고 효율적이다.
